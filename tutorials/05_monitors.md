## 这一章解决什么问题

仿真运行成功不等于实验完成。

如果没有设计记录方案，1000 个神经元运行 10 秒后，你可能只知道“程序没报错”，却无法回答：

- 膜状态怎样变化？
- 哪个神经元何时放电？
- 群体总体活动是增强还是减弱？
- 事件发生瞬间状态是多少？

Brian2 不会自动保存所有数据，因为那会占用巨大内存。你必须在运行前明确：

> 我的问题是什么，因此需要记录什么？

本章比较三种监视器：

| Monitor | 记录对象 | 典型问题 |
|---|---|---|
| `StateMonitor` | 连续状态 | v 如何随时间变化？ |
| `SpikeMonitor` | 离散脉冲 | 谁在何时放电？ |
| `PopulationRateMonitor` | 群体瞬时率 | 整体活动何时增强？ |

## 1. 本章神经元为什么不断波动

方程：

```text
dv/dt = (1.05 + 0.15*randn() - v)/(10*ms)
```

组成：

- 1.05：平均驱动略高于阈值 1。
- `-v`：漏衰减。
- `0.15*randn()`：每次求值时加入高斯随机波动。

这里使用 `randn()`，而上一章使用连续白噪声符号 `xi`。

两者都能引入随机性，但建模语义不同：

- `xi` 是随机微分方程中的白噪声项，具有明确连续时间尺度和单位规则。
- `randn()` 是方程求值时生成的标准正态随机数，其效果会更直接依赖更新方式与 dt。

本例使用 `randn()` 是为了快速制造不规则群体活动，重点放在 Monitor，而不是把它作为严格生物噪声模型模板。

## 2. 为什么使用 Euler 方法

```python
method="euler"
```

方程中每次更新都包含新的随机 `randn()`，不能使用上一章线性确定性方程的解析更新。

Euler 近似：

```text
v_next
= v_now
 + ((1.05 + 0.15*randn - v_now)/tau) * dt
```

这也意味着改变 dt 可能改变随机扰动的有效统计。若研究重点是严格随机过程，应使用明确的 SDE 写法并验证时间步收敛。

## 3. StateMonitor：连续轨迹的采样器

代码：

```python
state = StateMonitor(
    neurons,
    "v",
    record=[0, 1, 2],
)
```

它只记录 0、1、2 号神经元的 v。

为什么不记录全部 50 个？

假设：

- 记录 50 个神经元。
- 仿真 10 秒。
- dt=0.1 ms。
- 每个值使用 8 字节浮点数。

数据点数量约为：

```text
50 * 10 second / 0.0001 second
= 5,000,000 个值
```

仅一个变量原始数据就约 40 MB，还不含时间数组和对象开销。

当模型变大时，“默认全记录”很快不可行。

## 4. `record` 参数到底选择什么

常见写法：

```python
record=True
```

记录全部神经元。

```python
record=[0, 1, 2]
```

只记录指定索引。

```python
record=False
```

不记录任何索引，通常没有意义，除非对象配置后再调整。

对于大群体，推荐先问：

- 是否只需要代表性神经元？
- 是否只需要末值？
- 是否能降低记录采样频率？
- 是否应在线计算统计量，而不是保存全部轨迹？

## 5. StateMonitor 数据的形状

本例：

```python
state.v[0]
```

表示被记录的 0 号神经元轨迹。

`state.v` 第一维对应记录的神经元，第二维对应时间采样点。

如果 `record=[7, 20]`：

- `state.v[0]` 是 7 号神经元。
- `state.v[1]` 是 20 号神经元。

这里的行号是 Monitor 内部记录顺序，不一定等于原神经元编号。分析时要保留 record 列表。

## 6. SpikeMonitor：记录稀疏事件

代码：

```python
spikes = SpikeMonitor(neurons, variables="v")
```

基础记录包括：

- `spikes.t`：每次脉冲时间。
- `spikes.i`：对应神经元编号。
- `spikes.count`：每个神经元的脉冲数。
- `spikes.num_spikes`：总脉冲数。

事件数据通常比连续状态稀疏得多。

例如 50 个神经元平均 10 Hz、仿真 1 秒，大约记录：

```text
50 * 10 = 500 个脉冲
```

而 0.1 ms 采样的连续状态需要：

```text
50 * 10,000 = 500,000 个状态值
```

这就是事件驱动神经模型在数据表示上的优势之一。

## 7. 为什么 SpikeMonitor 还能记录变量

参数：

```python
variables="v"
```

表示每次 spike 事件发生时，额外记录该神经元的 v。

这叫事件触发采样：

- 不保存所有时刻的 v。
- 只保存事件附近调度点的 v。

它适合回答：

- 触发时 v 大约是多少？
- 不同事件时另一个适应变量是多少？
- 放电前的突触电流是否不同？

要注意，记录值对应事件调度中的特定时刻。阈值检测、Monitor 和 reset 的先后顺序会影响“记录到 reset 前还是 reset 后”。Brian2 的默认 SpikeMonitor 调度用于记录阈值事件，通常是在 reset 之前，但精确分析时应检查 `when` 和 `order`。

## 8. `count` 与 `num_spikes` 的区别

```python
spikes.count
```

是每个神经元的计数数组，例如：

```text
[6, 7, 6, 7, ...]
```

```python
spikes.num_spikes
```

是整个群体总数。

它们满足：

```text
sum(spikes.count) == spikes.num_spikes
```

但用途不同：

- 总数用于快速检查总体活动。
- count 分布用于判断神经元间异质性。

只报告总数可能掩盖“少数神经元承担了大部分活动”的情况。

## 9. PopulationRateMonitor 在计算什么

代码：

```python
rate = PopulationRateMonitor(neurons)
```

它按时间步统计整个群体发生的脉冲数，并除以：

- 神经元数量。
- 时间步长度。

概念上：

```text
瞬时群体率
= 本时间箱脉冲数 / 神经元数 / 时间箱宽度
```

单位是 Hz。

如果 50 个神经元在 0.1 ms 的同一个时间箱内共有 1 个脉冲：

```text
rate = 1 / 50 / 0.0001 second
     = 200 Hz
```

这不表示某个神经元持续以 200 Hz 放电。它表示把一个非常短时间箱中的群体事件换算成每神经元每秒的瞬时尺度。

因此原始 rate 往往非常尖锐。

## 10. 为什么要平滑群体率

代码：

```python
smoothed = rate.smooth_rate(
    window="gaussian",
    width=5 * ms,
)
```

平滑把邻近时间点的信息加权平均，使趋势更容易观察。

高斯窗意味着：

- 越靠近当前时刻，权重越大。
- 越远，权重越小。
- width 控制平滑时间尺度。

width 太小：

- 曲线仍然尖锐。
- 容易看到随机波动。

width 太大：

- 曲线更平滑。
- 快速变化可能被抹掉。
- 峰值时刻可能看起来扩散。

平滑不是“去除错误”，而是选择观察时间尺度。应报告使用的窗类型和宽度。

## 11. 三层图分别回答什么

### 第一层：三条 v 轨迹

回答：

- 单个神经元状态是否规则？
- reset 在哪里发生？
- 不同神经元轨迹是否同步？

### 第二层：脉冲栅格图

回答：

- 哪些神经元更活跃？
- 是否存在同步脉冲带？
- 活动是否在时间上均匀？

### 第三层：平滑群体率

回答：

- 群体总体活动水平如何变化？
- 是否存在宏观节律或活动峰？

三者不能互相替代：

- 三条 v 不能代表全部 50 个神经元。
- 栅格图不直接显示亚阈值状态。
- 平滑率会丢失具体神经元身份。

## 12. 记录方案应该从研究问题倒推

如果问题是“阈值前 v 怎样变化”：

- 需要 StateMonitor。
- 可能还需要事件对齐分析。

如果问题是“连接后是否出现同步”：

- SpikeMonitor 和栅格图更关键。

如果问题是“刺激后群体活动提高多少”：

- PopulationRateMonitor 或时间窗内平均率更合适。

如果问题是“每个神经元的平均率分布”：

- 使用 SpikeMonitor.count，再除以仿真时长。

先记录再想问题，往往会导致数据很多但证据不足。

## 13. Monitor 也会影响性能和内存

仿真变慢时，不要只盯着神经元方程。

常见数据瓶颈：

- StateMonitor 记录太多神经元。
- 记录时间步太细。
- 同时记录太多变量。
- 长仿真全部保存在内存。
- 绘图处理超大数组。

性能分析中经常会看到 `statemonitor` 占据显著时间，这不是 Brian2 出错，而是你要求它搬运和保存大量数据。

## 14. 常见误区

### 误区一：PopulationRateMonitor 是每个神经元的放电率

它记录群体整体随时间变化的瞬时率。

### 误区二：平滑率是唯一真实的放电率

不同窗宽对应不同观察尺度，没有唯一正确的平滑曲线。

### 误区三：StateMonitor 的第 0 行永远对应神经元 0

它对应 record 列表中的第一个对象。

### 误区四：没记录的数据运行后还能随时取回

大多数历史轨迹若未通过 Monitor 记录，运行后无法重建。

### 误区五：Monitor 只是观察者，不影响资源

它不应改变模型动力学，但会消耗运行时间和内存。

## 本章小结

三类 Monitor 对应三个观察层次：

```text
StateMonitor          -> 连续微观状态
SpikeMonitor          -> 离散个体事件
PopulationRateMonitor -> 宏观群体活动
```

真正的实验设计不是“全部记录”，而是：

```text
问题 -> 所需证据 -> Monitor -> 数据分析
```

下一章会让突触权重随脉冲时序变化。届时 Monitor 不仅要记录网络输出，还要验证“学习规则是否真的改变了连接”。
